## Контекст проекта

Проект реализует ускоренное множественное выравнивание последовательностей (MSA)
с помощью нейросети, предсказывающей ширину и центр diagonal band для алгоритма
Нидлмана-Вунша (banded NW). Обучение и тестирование — на синтетических ДНК и белковых данных.

Результаты последнего прогона (`FINAL_REPORT.md`) выявили три независимые проблемы.
Устраняй их **строго по порядку** — проблема 1 может быть причиной проблем 2 и 3.

## ПРОБЛЕМА 1 (приоритет: критический)

### Описание
SP-score нашего метода на синтетических данных ниже ожидаемого.

### Где искать

Файлы, связанные с оценкой MSA качества:
- модуль, вычисляющий SP-score / TC-score
- код, сопоставляющий `predicted_alignment` с `reference_alignment` по seq ID

### Что проверить

1. **Совпадают ли seq ID** в предсказанном выравнивании и в эталоне.

2. **Добавь диагностический вывод** в функцию вычисления SP-score:

```python
def compute_sp_score(predicted: dict, reference: dict) -> float:
    pred_ids = set(predicted.keys())
    ref_ids = set(reference.keys())

    # ДИАГНОСТИКА — убрать после починки
    missing_in_pred = ref_ids - pred_ids
    extra_in_pred = pred_ids - ref_ids
    if missing_in_pred or extra_in_pred:
        print(f"[SP WARNING] ID mismatch!")
        print(f"  Missing in predicted: {missing_in_pred}")
        print(f"  Extra in predicted:   {extra_in_pred}")
        return 0.0  # сейчас возвращает 0 вместо -1 — это маскирует ошибку

    # ... остальной код
```

3. **Проверь нормализацию ID** — часто проблема в strip / split:

```python
# Плохо — может обрезать нужные символы или оставить лишние
seq_id = line.split()[0].strip(">").strip()

# Хорошо — явно убираем только ">"
seq_id = line.split()[0].lstrip(">")
```

### Критерий успеха
После исправления запусти эксперимент `msa_quality`. SP-score для `Neural_band`
должен быть **> 0.50** (даже без finetuning на белках).
Если SP всё ещё < 0.20 — domain shift реален, переходи к Проблеме 1б.

## ПРОБЛЕМА 2 (приоритет: высокий)

### Описание
`Neural_+_refine` даёт SP = **0.068** — хуже, чем `Neural_band` (0.130).
Итеративное уточнение не улучшает, а ухудшает выравнивание.

### Где искать

Функция/класс, реализующий итеративное уточнение (refinement) MSA.
Обычно это цикл, который поочерёдно убирает одну последовательность,
перевыравнивает её к профилю остальных и вставляет обратно.

### Что проверить

1. **Добавь логирование SP после каждой итерации:**

```python
def refine_msa(alignment, max_iter=10):
    best_sp = compute_sp_score(alignment, reference)
    best_alignment = alignment.copy()

    for i in range(max_iter):
        alignment = _one_refinement_step(alignment)
        sp = compute_sp_score(alignment, reference)
        print(f"[REFINE] iter={i}, SP={sp:.4f}")  # ДИАГНОСТИКА

        if sp > best_sp:
            best_sp = sp
            best_alignment = alignment.copy()
        # ВАЖНО: не обновлять alignment если SP упал — сейчас этого может не быть

    return best_alignment  # возвращать лучшее, не последнее
```

2. **Проверь критерий остановки** — скорее всего его нет совсем, и refinement
   продолжается даже когда качество падает.

3. **Проверь, не перезаписывается ли профиль** до того, как все
   последовательности в итерации перевыровнены. Это классический баг:

```python
# ПЛОХО — профиль обновляется сразу, следующая seq выравнивается к уже изменённому профилю
for seq in alignment:
    profile = build_profile(alignment)  # профиль внутри цикла — баг!
    alignment[seq] = realign(seq, profile)

# ХОРОШО — профиль строится один раз до начала итерации
profile = build_profile(alignment)
new_alignment = {}
for seq in alignment:
    new_alignment[seq] = realign(seq, profile)
alignment = new_alignment
```

### Критерий успеха
SP для `Neural_+_refine` должен быть **≥ SP для `Neural_band`** (сейчас 0.130).
Ожидаемый прирост после починки: +0.02–0.10.

## ПРОБЛЕМА 3 (приоритет: средний)

### Описание
Four Russians `hit_ratio = 0.000` на всех 300 парах. Таблица растёт до 17 МБ,
но ни один блок не используется повторно.

### Где искать

Модуль Four Russians — функции заполнения таблицы (`store_block`) и поиска
(`lookup_block`). Параметр `t` (размер блока).

### Что проверить

1. **Выведи несколько примеров поиска** — что ищем vs что лежит в таблице:

```python
def lookup_block(self, block_key):
    if block_key in self.table:
        self.hits += 1
        return self.table[block_key]
    else:
        self.misses += 1
        # ДИАГНОСТИКА — убрать после починки
        if self.misses <= 5:
            sample_keys = list(self.table.keys())[:3]
            print(f"[FR MISS] Looking for: {block_key}")
            print(f"[FR MISS] Sample keys in table: {sample_keys}")
        return None
```

2. **Уменьши параметр `t`** — попробуй `t=2` или `t=3` (сейчас, вероятно, больше):

```python
# Меньший t → больше уникальных блоков совпадает → выше hit_ratio
FOUR_RUSSIANS_T = 2  # было: предположительно 4 или 5
```

3. **Проверь нормализацию ключей** — блок должен хэшироваться только по
   содержимому, без учёта абсолютной позиции в матрице:

```python
# ПЛОХО — позиция входит в ключ, совпадений никогда не будет
block_key = (row_start, col_start, tuple(block_values))

# ХОРОШО — только содержимое
block_key = tuple(block_values)
```

### Критерий успеха
После 300 пар `hit_ratio` должен быть **> 0.3**. Значение > 0.5 — отлично.

## ПРОБЛЕМА 4 (приоритет: низкий)

### Описание
`recall@1x = 0.513` при обучении — ниже желаемого порога 0.80.
Loss почти не снижается (2.46 → 2.32 за 5 эпох).

### Возможные причины и решения

| Симптом | Вероятная причина | Что сделать |
|---------|-------------------|-------------|
| Loss не падает | Learning rate слишком большой или маленький | Попробовать LR scheduler (cosine или reduce on plateau) |
| recall@1x низкий, но recall@1.5x = 0.987 | Модель консервативна, предсказывает чуть узкую полосу | Добавить в loss штраф за underestimate сильнее, чем за overestimate |
| Loss застрял на 2.3x | Градиенты затухают | Проверить нормализацию входных признаков, добавить batch norm |

**Минимальное изменение — asymmetric loss:**

```python
def band_loss(pred_hw, true_hw, alpha=2.0):
    """
    alpha > 1 штрафует underestimate сильнее.
    Underestimate опасен (неправильное выравнивание),
    overestimate просто медленнее.
    """
    diff = pred_hw - true_hw
    loss = torch.where(diff < 0, alpha * diff**2, diff**2)
    return loss.mean()
```

### Критерий успеха
`recall@1x` на валидации **> 0.75** после дополнительного обучения.
